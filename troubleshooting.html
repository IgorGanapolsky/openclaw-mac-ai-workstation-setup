The six real causes (in order of how often they actually break things)

Symptom 1 — "Computer Use plugin unavailable" even after reinstall The bundled plugin exists in /Applications/Codex.app/Contents/Resources/plugins/openai-bundled but the active Codex home isn't pointed at it. Reinstalling doesn't fix this; the plugin cache lives in your user directory and the install path didn't touch it. 
codex mcp list
ls -la ~/.codex/plugins/cache/openai-bundled/computer-use/*/.codex-plugin/plugin.json
ls -la ~/.codex/plugins/cache/openai-bundled/computer-use/*/.mcp.json
codex plugin marketplace add /Applications/Codex.app/Contents/Resources/plugins/openai-bundled
codex mcp list

If the second codex mcp list still doesn't show computer-use, you have a stale Codex home or isolated profile — see Symptom 5.

Symptom 2 — Screen Recording and Accessibility both granted, app still denies approval macOS TCC tracks permission by bundle identifier, not by app name. If you ran Codex from a different bundle path (DMG copy vs. App Store vs. helper signed differently), permissions are recorded for one bundle and the running process has a different one. 
# Confirm which bundle is actually running
ps aux | grep -i codex | grep -v grep
codesign -dv --verbose=2 /Applications/Codex.app 2>&1 | grep -E "Identifier|TeamIdentifier"
# Compare to what TCC has approved (will prompt for Full Disk Access)
sqlite3 ~/Library/Application\ Support/com.apple.TCC/TCC.db \
  "SELECT client, auth_value FROM access WHERE service IN ('kTCCServiceScreenCapture','kTCCServiceAccessibility');"

Mismatch between the running bundle ID and the TCC-approved bundle ID is the actual cause. The fix is to fully remove the unsigned/old copy and re-grant permissions to the canonical bundle.

Symptom 3 — OpenRouter "fallback" silently routes you back to the broken primary Gateway fallback config keeps the primary model's threshold values when escalating to the fallback, so the fallback hits the same compression or context limit and fails identically. The agent looks like it tried two providers; it actually tried one provider twice. 
# Inspect the active route used for your last failing call
openclaw route show --last
openclaw route show --agent computer-use
# Look for "threshold_inherited_from_primary": true

Fix: define separate threshold/context configs per route, or split into a dedicated agent variant (this is what the working setup here does — main agent on Claude Sonnet via OpenRouter, dedicated computer-use agent on Codex runtime).

Symptom 4 — Agent works in Zed terminal, breaks the moment you launch it from cmux / Warp / a host app Apple Events from a child process inherit the parent's TCC posture, not their own. A tool running inside cmux has cmux's permissions, not Codex's, even though Codex started the tool. This produces AppleScript error -1743 ("Not allowed to send Apple events") with no other diagnostic. 
# Verify which parent is granting events
ps -o ppid,command -p $$
# Grant Automation permission to the actual parent (cmux/Warp/iTerm) in:
# System Settings > Privacy & Security > Automation
Symptom 5 — Stale OAuth tokens or multiple Codex homes blocking the runtime Codex stores auth under one $CODEX_HOME; if you have an old shell that exported a different one, or an installer that wrote to a per-user home while you're running as a different user (think of the macOS "Codex updated under a different user" class of bug), the runtime that's answering text is not the same runtime that owns the Computer Use plugin. 
printenv | grep -i codex
ls -la ~/.codex ~/Library/Application\ Support/Codex 2>/dev/null
# Pick one canonical CODEX_HOME and remove the others from your shell rc
Symptom 6 — Same OpenClaw TUI session reused across unrelated projects OpenClaw and otui reuse a session pool keyed by terminal, not by working directory. Open the same agent in two project dirs and it inherits the other project's compaction state, plan, and memory — producing answers that reference files from the wrong project. 
# Confirm session reuse
openclaw session ls
# Use a per-directory wrapper that hashes $PWD into the session id
# (or run with `openclaw --session-key "$(pwd | shasum | cut -c1-12)"`)

The eight real causes (in the order they most often break things)

Cause 1 — Message Content Intent disabled in the Discord Developer Portal Discord requires the bot's application to explicitly enable the "Message Content Intent" privileged intent before it will deliver actual message text. Without it, Discord delivers the event but redacts the content field — the Claude Code plugin sees activity, gets nothing to respond to, and stays silent. Open https://discord.com/developers/applications, select the app, go to Bot, toggle Message Content Intent on, save, restart Claude Code with --channels.
Cause 2 — Missing --channels flag on startup The plugin does not auto-load. Launch with the flag attached or it stays dormant: 
claude --channels plugin:discord@claude-plugins-official
Confirm the flag is on the running process: 
ps -axo pid,command | grep -i claude | grep -i channels
If the result is empty, the running session has no channels plugin attached regardless of what the TUI says.
Cause 3 — Pairing flow done in the wrong order Pairing requires two halves in a specific order: (a) you DM the bot from your Discord account, the bot replies with a fresh pairing code; (b) you run /discord:access pair YOUR_CODE in Claude Code immediately, while that code is still valid. Running the pair command first, or letting the code expire, leaves the access record incomplete and the bot stays mute on every later message.
Cause 4 — Bot and your Discord account don't share a server Discord drops DMs silently if the two parties don't share at least one server. Easy to miss when you generate an OAuth invite link, send the bot to a server you've since left, then DM the bot from your main account. The bot is "online" but DMs to it never arrive. Fix: confirm at least one shared server, then re-pair.
Cause 5 — Session resumption silently detaches the plugin Claude Code session resume rehydrates the conversation state from disk, but the Discord WebSocket connection isn't part of that persisted state. After a resume, the session looks fine in the TUI; the bot is functionally disconnected. There is no error. Fix: don't resume — start a fresh session with --channels when you want Channels active.
Cause 6 — Channel notification routing fails to reach the session The plugin reports connected in /mcp, the Discord gateway TCP socket is established, but inbound messages never become user turns. Confirmed pattern from multiple open reports (see anthropics/claude-code #36837, #36657, #37287). To isolate whether the plugin itself emits the notification, run the plugin's standalone server (bun run start against the same Discord token in the plugin's working dir) — if standalone receives the DM and emits a notifications/claude/channel JSON-RPC frame, the bug is in how Claude Code consumes that notification, not in the plugin's Discord gateway. Workaround: restart with --channels (don't resume), or pre-grant tool permissions so prompts don't interrupt the loop.
Cause 7 — Permission-relay deadlock after Allow/Deny prompt After approving (or denying) a tool-use permission prompt in the TUI, the resolution isn't always relayed back to the Discord event loop — the session waits for an approval that already happened. Documented in anthropics/claude-code#43405. To isolate whether the relay path is the cause vs. a different failure, run a one-off test session with --dangerously-skip-permissions in a trusted local environment and see if the symptom disappears. (Don't ship that flag in real use — it bypasses every tool-use approval.) If yes, the relay path is the cause and the durable fix is upstream; pre-grant the tool classes you'll use as a workaround so prompts don't fire mid-Discord-exchange.
Cause 8 — tengu_harbor GrowthBook flag gates channels off in 2.1.114+ on personal Max Claude Code CLI 2.1.114 added a server-side feature-flag gate (tengu_harbor, evaluated via GrowthBook) on channels registration. For personal Max (Stripe-subscription) accounts the flag returns false at runtime even when ~/.claude.json shows cachedGrowthBookFeatures.tengu_harbor: true — the live evaluation overrides the cache. Symptom is a startup print of --dangerously-load-development-channels ignored followed by Channels are not currently available, and MCP notifications/claude/channel pushes silently dropped despite the server-side POST succeeding.

Confirmed by decompilation in anthropics/claude-code#50903 (credit: @cxm-ghu). The gate in 2.1.114: 
function IK_(serverName, serverCaps) {
  if (!serverCaps?.experimental?.["claude/channel"])
    return { action: "skip", kind: "capability" };
  if (!TPH())  // ← gate; returns false for personal Max
    return { action: "skip", kind: "disabled",
             reason: "channels feature is not currently available" };
  // ...team/enterprise channelsEnabled check follows
}
function TPH() { return h_("tengu_harbor", false); }
The channelsEnabled managed setting at /Library/Application Support/ClaudeCode/managed-settings.json does not apply on Max — that branch is gated behind isTeamOrEnterprise in the same function. There is no documented user-side override on personal Max today.

To check if you're hitting this specifically (vs Cause 1-7): launch with claude --dangerously-load-development-channels … and grep startup output for Channels are not currently available. Presence of that string = this is your cause and none of the local-config fixes above will resolve it.

Mitigation: pin Claude Code to 2.1.104 (last version where the flag wasn't gated), or run the Channels plugin server standalone and bridge via your own MCP setup. Until Anthropic clarifies the rollout, those are the only working paths.

Why most of these never show an error

Causes 1, 3, 4, 5, 6 produce a session that looks healthy in both Discord and Claude Code. No error message, no crash, no log entry that points at the failure. You're debugging absence-of-output, not a stack trace. Cause 8 is the loud exception — it prints Channels are not currently available at startup, which actually makes it the easiest to distinguish if you know to look. That's why the standard "uninstall, reinstall, restart" approach almost never resolves them — the failure surface is in configuration, topology, or server-side feature flags, not in the code path that errors out loudly.
